# 模块规则

## 划分原则

- **按业务功能**：每个模块对应具体业务领域
- **高内聚低耦合**：内部功能相关，模块间依赖最小
- **可独立部署**：支持独立开发和部署
- **可测试性**：易于单元测试和集成测试

## 命名规范

- **模块名**：小写字母和下划线，如 `user_management`
- **文件**：小写字母和下划线
- **类名**：PascalCase，如 `UserService`
- **函数**：camelCase，如 `getUserInfo`

## 模块结构

### 后端

```
module_name/
├── api/          # API路由
├── services/     # 业务逻辑
├── models/       # 数据模型
├── schemas/      # 数据验证
└── utils/        # 工具函数
```

### 前端

```
module_name/
├── components/   # 组件
├── pages/        # 页面
├── services/     # API调用
├── store/        # 状态管理
└── utils/        # 工具函数
```

## 依赖管理

- **内部依赖**：通过接口调用，避免直接依赖实现
- **外部依赖**：明确版本，使用包管理工具
- **依赖注入**：提高可测试性

## 接口规范

- **API接口**：RESTful规范，HTTP方法和状态码
- **内部接口**：清晰函数签名和返回类型
- **错误处理**：统一机制，标准错误格式

## 测试要求

- **单元测试**：模块内部功能
- **集成测试**：模块间交互
- **端到端测试**：完整业务流程

## 文档

- **API文档**：Swagger/OpenAPI生成
- **模块说明**：功能、接口和使用方法
- **变更记录**：版本变更和功能调整

## 新增流程

1. **需求分析**：明确功能和范围
2. **设计**：模块结构和接口
3. **实现**：按设计实现功能
4. **测试**：编写测试用例
5. **文档**：更新模块文档
6. **集成**：集成到项目中
7. **部署**：部署并监控

## 维护

- **代码审查**：定期审查，确保质量
- **性能优化**：监控并优化
- **安全更新**：及时修复漏洞
- **版本管理**：语义化版本
